## Resumen ejecutivo

El SDK no introduce vectores remotos directos. Lo que sí mete es **superficie de filtración**: claves, credenciales SOL y tokens OAuth2 viven en memoria sin envoltura de redacción, y un parser XML configurado en modo permisivo. Si el caller activa `logging.DEBUG` (cosa común para depurar prod) o hace `print(bundle)` (más común aún en tests), termina vertiendo material que debió quedar oculto.

El hallazgo de mayor costo si se posterga es `CertBundle.__repr__` (#A1). Vive en la API pública del paquete, así que cambiarlo después de tener usuarios PyPI implica decidir entre "la key se filtra en todo log/test" y "romper `repr()` para callers que ya dependen del formato". ## Detalle

### A1 — `CertBundle.__repr__` default filtra la clave privada

**Severidad:** Alto.
**Estado:** Resuelto en 0.4.1 (working tree, commit pendiente).
**Archivo:** `packages/core/src/sunat_py/security/cert_loader.py:20-25` (+ `:53-57`).

`CertBundle` es un `@dataclass(frozen=True)` con cuatro campos públicos, dos de ellos sensibles:

```python
@dataclass(frozen=True)
class CertBundle:
    private_key: RSAPrivateKey
    certificate: Certificate
    cert_pem: bytes
    key_pem: bytes
```

La `key_pem` se genera con `encryption_algorithm=serialization.NoEncryption()` (línea 56), entonces es PEM en claro. El `__repr__` que Python genera por default para un dataclass lo dump completo. Cualquier de estas líneas mete la clave privada en un log o stdout:

```python
print(bundle)
logger.info("loaded cert: %s", bundle)
assert bundle.serial_hex == expected, bundle   # AssertionError lleva repr(bundle)
raise RuntimeError(f"cert mal cargado: {bundle}")
```

Vale lo mismo para Sentry breadcrumbs, frames de pytest, o cualquier handler de errores que serialice variables locales.

`CertBundle` está exportado en `sunat_py/__init__.py:8-12`, así que el caller lo recibe directo del top-level del paquete.

**Vector concreto:** un usuario del SDK que adopte una práctica defensiva razonable ("loggeo el objeto que cargué para verificar"), termina con la clave privada del contribuyente en su pipeline de logs centralizado.

**Fix conceptual:** sobreescribir `__repr__` y `__str__` en el dataclass.

```python
def __repr__(self) -> str:
    return f"CertBundle(common_name={self.common_name!r}, serial={self.serial_hex}, key=<redacted>)"
```

Opcional pero recomendable: marcar `private_key` y `key_pem` con `field(repr=False)` ya da el mismo efecto sin tocar `__repr__`. No romper `serial_hex`/`common_name` que son seguros y útiles para debugging.

### A2 — `xml_huge_tree=True` + `strict=False` en zeep settings

**Severidad:** Alto.
**Estado:** Resuelto en 0.4.1. Validado contra SUNAT beta (F001-1 accepted code 0) y prod (round-trip OK, fault 1033 parseado limpio con strict=True). El fallback a `strict=False` que se dejaba abierto en el fix conceptual no fue necesario.
**Archivo:** `packages/core/src/sunat_py/sunat/client.py:73`.

```python
zeep_settings = ZeepSettings(strict=False, xml_huge_tree=True)
```

Dos flags relajados de una sola línea:

- `xml_huge_tree=True` deshabilita la protección de lxml contra árboles "huge", que es lo que defiende contra **Billion Laughs** y **quadratic blowup** (entity expansion bombs). El parser deja de imponer límites de profundidad / nodos.
- `strict=False` hace que zeep no rechace respuestas que no validan contra el WSDL. Cualquier campo de más, cualquier tag inesperado, pasa.

La protección que queda es solo TLS hacia SUNAT.

**Vector concreto:** un CDR que SUNAT (legítimamente) devuelve con varios MB infla a GB en árbol parseado y agota la RAM del worker. Un escenario menos hipotético: si en algún futuro el código pasa por un proxy MITM (debugging interno, mitmproxy en dev), un payload con entidades anidadas cuelga el proceso o lo OOM-ea. Una `requests.Session` con un proxy local mal configurado bastaría para reproducirlo.

`huge_tree` no se necesita salvo que SUNAT en serio mande XMLs grandes — los CDR reales son de pocos KB. La opción más sana es dejar el default y, si en algún momento aparece un CDR legítimo grande, imponer un cap de bytes explícito antes de `etree.fromstring` (10 MB es de sobra).

**Fix conceptual:**

```python
zeep_settings = ZeepSettings(strict=True)   # sin xml_huge_tree
```

Si SUNAT alguna vez devuelve algo enorme y zeep lo rechaza, agregar un parser custom con `XMLParser(huge_tree=False, resolve_entities=False)` y un guard de tamaño antes.

### A3 — Credenciales OAuth2 en form body sin masking

**Severidad:** Alto.
**Estado:** Resuelto en 0.4.1. Docstring + helper opt-in `sunat_py.security.install_log_redactor()`. Tests en `tests/unit/test_log_redactor.py`.
**Archivo:** `packages/core/src/sunat_py/sunat/gre_client.py:47-58`.

```python
resp = requests.post(
    url,
    data={
        "grant_type": "password",
        "scope": "https://api-cpe.sunat.gob.pe",
        "client_id": client_id,
        "client_secret": client_secret,
        "username": f"{ruc}{username}",
        "password": password,
    },
    timeout=20,
)
```

El body viaja como `application/x-www-form-urlencoded` con cuatro secretos en cleartext: `client_id`, `client_secret`, `username` (RUC + sufijo SOL), `password` (clave SOL). Sobre TLS es OK en el wire, el problema es local: `urllib3.connectionpool` loggea el body completo cuando alguien hace `logging.basicConfig(level=logging.DEBUG)` o `logging.getLogger("urllib3").setLevel(logging.DEBUG)`. Es la primera receta que aparece en cualquier respuesta de StackOverflow para "debug requests in Python".

**Vector concreto:** dev en producción dice "no entiendo qué le pasa al GRE", activa logging DEBUG, reproduce el envío, el output incluye `password=...` en cleartext, y ese log termina en CloudWatch / Vercel logs / archivo local.

**Fix conceptual:** dos partes.

1. Documentar en el docstring de `get_gre_token` que el caller **no debe** habilitar logging DEBUG de urllib3 en producción.
2. Opcional: registrar un `logging.Filter` en el logger de urllib3 que reemplace `password=...` y `client_secret=...` con `<redacted>` cuando el SDK se importa. Es opt-in vía función helper (`sunat_py.security.install_log_redactor()`) — no se debe instalar global por respeto al caller.

### A4 — Bearer token expuesto en `headers` dict

**Severidad:** Alto.
**Estado:** Resuelto en 0.4.1. Mismo redactor opt-in de A3 enmascara `Bearer ...`. Docstring de `send_gre` advierte al caller.
**Archivo:** `packages/core/src/sunat_py/sunat/gre_client.py:80-83`.

```python
headers = {
    "Authorization": f"Bearer {token}",
    "Content-Type": "application/json",
}
```

El token vuelve a estar plano en una estructura que cualquier wrapper del caller serializa con `repr()` por costumbre. `requests.HTTPError` por default **no** incluye los headers en su `str()`, pero hooks de logging (Sentry, librerías de observabilidad estilo `logfire`) sí los capturan completos.

Adicional: `get_gre_token` (línea 60) retorna el token como `str` plano para que el caller decida persistencia. Si lo persiste en BD sin cifrar, queda accesible al primer leak de la BD. Eso es responsabilidad del caller, pero el SDK debe avisar.

**Vector concreto:** mismo perfil que A3. Token bearer en logs de prod o en breadcrumbs de Sentry.

**Fix conceptual:**

1. Docstring de `get_gre_token` con párrafo explícito: "Este token autoriza envíos GRE en nombre del RUC durante su TTL. Trátalo como secret: no lo escribas a disco sin cifrar, no lo loggees, no lo expongas en respuestas HTTP del caller."
2. Mismo filtro de logging del A3 también enmascara `Authorization: Bearer ...` si se opta por instalarlo.

### M1 — `except Exception: pass` silencia errores parseando CDR status 99

**Severidad:** Medio.
**Archivo:** `packages/core/src/sunat_py/sunat/client.py:170-195`.

```python
if status_code == "99":
    if content:
        try:
            cdr_xml = unpack_cdr(content)
            code, description = _parse_cdr(cdr_xml)
            return ("done", status_code, SunatResult(status="rejected", code=code, ...))
        except Exception:
            pass
    return ("done", status_code, SunatResult(status="rejected", code="99",
            description="ticket procesado con errores", cdr_xml=b""))
```

Si el ZIP del CDR llega corrupto o el XML adentro está malformado, el SDK retorna un resultado de "rejected" sin trazabilidad: el caller no puede distinguir entre "SUNAT rechazó con un CDR válido cuyo detalle no pude leer" y "el blob que llegó era basura".

**Vector concreto:** no es un exploit, es un blindspot. Tampering en tránsito (poco probable con TLS sano), bug en SUNAT (más probable de lo que parece), o un cambio futuro en el formato de respuesta — todos se ven igual al caller, y eso dificulta el incident response.

**Fix conceptual:** capturar excepción específica y propagarla en el resultado.

```python
except (zipfile.BadZipFile, lxml.etree.XMLSyntaxError) as exc:
    return ("done", status_code, SunatResult(
        status="rejected", code="99",
        description=f"ticket procesado con errores; CDR ilegible: {exc!r}",
        cdr_xml=b"",
    ))
```

### M2 — Passphrase del PFX como `str` (no zeroizable)

**Severidad:** Medio.
**Archivo:** `packages/core/src/sunat_py/security/cert_loader.py:37, 67`.

```python
def load_cert_from_pfx(pfx_bytes: bytes, password: str | None = None) -> CertBundle:
    pwd_bytes = password.encode("utf-8") if password else None
```

Python no permite zeroizar strings — quedan en el intern pool y/o en el heap hasta que el GC los recoja, y para entonces pueden haber pasado por varios frames. En memory dumps (core dump, post-mortem) la passphrase aparece en cleartext.

Es defensa en profundidad: para que importe en serio, el atacante ya tiene RCE o lectura de proceso, escenario en el que también tiene la `key_pem` ya descifrada de A1. Pero es un grado más de exposición que no cuesta nada bajar.

**Fix conceptual:** documentar en el docstring que el caller debe pasar la passphrase justo antes de la carga y dejar la referencia al GC (`pwd = None` después de `load_cert_from_pfx`). Aceptar también `bytes`:

```python
def load_cert_from_pfx(pfx_bytes: bytes, password: str | bytes | None = None) -> CertBundle:
```

Y, dentro de la función, sobrescribir el buffer local si llega como `bytearray`. No es perfecto en CPython pero acerca el modelo a "borra después de usar".

### M3 — `unpack_cdr` sin caps de tamaño ni de entries

**Severidad:** Medio.
**Archivo:** `packages/core/src/sunat_py/sunat/packager.py:17-33`.

```python
with zipfile.ZipFile(io.BytesIO(zip_bytes), "r") as zf:
    for name in zf.namelist():
        if name.endswith(".xml"):
            return zf.read(name)
```

Sin `max_size`, sin contar entries, sin chequear ratio compresión. `zf.read(name)` deserializa el archivo entero a memoria. Un CDR malicioso de 1 KB que descomprime a 4 GB OOM-ea el proceso.

El ZIP viene de SUNAT, así que la confianza es alta — pero, igual que en A2, el modelo de amenaza incluye el caso MITM en dev y el caso "SUNAT mandó algo raro". El cap explícito es barato.

**Fix conceptual:**

```python
MAX_CDR_BYTES = 10 * 1024 * 1024  # 10 MB es 1000x lo razonable
with zipfile.ZipFile(io.BytesIO(zip_bytes), "r") as zf:
    infos = [i for i in zf.infolist() if i.filename.endswith(".xml")]
    if not infos:
        raise RuntimeError("CDR ZIP no contiene XML")
    info = infos[0]
    if info.file_size > MAX_CDR_BYTES:
        raise RuntimeError(f"CDR XML excede {MAX_CDR_BYTES} bytes (file_size={info.file_size})")
    return zf.read(info)
```

`info.file_size` es metadata de la entry, se chequea sin descomprimir.

### M4 — `UsernameToken` retenido en cliente zeep sin docstring de advertencia

**Severidad:** Medio.
**Archivo:** `packages/core/src/sunat_py/sunat/client.py:51-79`.

```python
wsse = UsernameToken(f"{ruc}{username}", password)
return Client(wsdl=str(wsdl_path), wsse=wsse, transport=transport, settings=zeep_settings)
```

El cliente zeep retiene el `UsernameToken` en su estado interno mientras viva. `Client` no tiene `__repr__` masking propio para wsse, y el comportamiento exacto depende de la versión de zeep. Si el caller hace `pickle.dumps(client)` (escenario obvio: serializar a cache distribuido) o si activa logging DEBUG de zeep, el password viaja.

**Vector concreto:** caller intenta cachear el `zeep.Client` en Redis para evitar el costo de parsear WSDL en cada request. El `pickle.dumps(client)` incluye la password. Termina como bytes en Redis sin cifrar.

**Fix conceptual:** dos cambios mínimos en docstring de `build_zeep_client`:

> El cliente retiene las credenciales SOL en su estado interno. **No serializarlo** (pickle, joblib) ni loggear su repr. Para cachear entre requests usar referencia en memoria del proceso; para invalidar credenciales rotadas, descartar el cliente y construir uno nuevo.

Opcional: exponer una función `clear_credentials(client)` que setee `client.wsse = None` para el caso "ya autentiqué y solo quiero el transport".

### B1 — Sin pinning de certificado TLS de SUNAT

**Severidad:** Bajo.
**Archivos:** `sunat/client.py:72`, `sunat/gre_client.py:47, 87, 100`.

`verify=True` default delega al trust store del sistema. SUNAT publica certs desde CAs públicas conocidas, así que el riesgo realista es bajo: requiere que una CA del trust store del sistema se haya comprometido **y** que el atacante pueda redirigir el tráfico (DNS poisoning, BGP hijack, proxy malicioso en la red corporativa del caller).

**Fix conceptual:** defensa en profundidad — exponer `verify` opcional como parámetro y permitir al caller pasar un bundle propio (`/etc/ssl/sunat-ca.pem`) si lo necesita. No cambiar el default. Pinning hardcoded en el SDK no escala (los certs de SUNAT rotan y el SDK quedaría desactualizado).

### B2 — `filename_base` entra crudo a URL y a `zipfile.writestr`

**Severidad:** Bajo.
**Archivos:** `sunat/gre_client.py:76, 86`, `sunat/packager.py:13`.

```python
# gre_client.py
zip_name = f"{filename_base}.zip"
send_url = f"{GRE_API_HOST}/v1/contribuyente/gem/comprobantes/{filename_base}"

# packager.py
zf.writestr(f"{filename_base}.xml", xml)
```

Si el caller pasa `filename_base="20XXXXXXXXX-01-F001-1/../etc"`, la URL queda alterada antes de salir del proceso. SUNAT casi seguro lo rechaza, pero un cliente HTTP intermedio (proxy de logs, WAF) puede comportarse raro. No es explotación remota, es "no sanitizar lo que el caller pasa".

**Fix conceptual:** validar con regex al inicio de `send_gre` y `pack_invoice`. SUNAT formato canónico: `^\d{11}-\d{2}-[A-Z]\d{3}-\d+$` para CPE.

```python
import re
_FILENAME_BASE_RE = re.compile(r"^\d{11}-\d{2}-[A-Z]\w{3}-\d+$")
if not _FILENAME_BASE_RE.match(filename_base):
    raise ValueError(f"filename_base no cumple formato SUNAT: {filename_base!r}")
```

### B3 — Sin lockfile committed

**Severidad:** Bajo.
**Ubicación:** repo root, `packages/core/`.

No hay `uv.lock`, `poetry.lock`, ni `requirements-lock.txt`. Las deps están pineadas por major (`lxml>=5.3,<6`, etc.), pero la versión exacta resuelta depende del momento del `pip install`. Para un paquete que firma documentos tributarios, la falta de lockfile reduce reproducibilidad de builds: una vulnerabilidad transitiva entra silenciosa en el siguiente release sin que nadie note.

**Fix conceptual:** `uv lock` o `pip-compile` para generar un `requirements.txt` reproducible en CI y comparable entre releases.

### B4 — Publicación a PyPI manual, sin trusted publishing

**Severidad:** Bajo.
**Archivo:** `.github/workflows/release.yml`.

El workflow actual al recibir un tag `v*.*.*` solo crea un GitHub Release con las notas del CHANGELOG. **No** publica a PyPI. La publicación es manual (`twine upload` desde la laptop) — eso significa:

- Sin atestación SLSA / Sigstore.
- Sin OIDC trusted publishing.
- Si Luis usa un API token, vive en su Keychain/keyring y rota a mano.
- Riesgo de subir un build sucio (con archivos del working tree) o de subir versión equivocada por typo.

**Fix conceptual:** agregar job al workflow que use [PyPI trusted publishing con OIDC](https://docs.pypi.org/trusted-publishers/). El workflow autentica con GitHub via OIDC, sin token; PyPI registra el publisher y rechaza uploads desde otros entornos. Tres pasos:

1. En pypi.org, registrar `userDKX/SIS-Facturador` como publisher trusted del proyecto `sunat-py`.
2. En `release.yml` agregar `permissions: id-token: write` al job.
3. Reemplazar el `twine upload` manual por `pypa/gh-action-pypi-publish@release/v1`.

### I1 — XXE en builders neutralizado por autoescape de Jinja2

**Severidad:** Info (no es bug).
**Archivos:** `ubl/builder.py:144-149` + 6 sitios `:245, 282, 318, 348, 401, 442`.

En seis builders el SDK invoca `etree.fromstring(rendered.encode("utf-8"))` para validar well-formedness post-render. Sin más contexto eso suele ser un finding XXE clásico (lxml sin `defusedxml`).

**Pero** el entorno Jinja2 está configurado así:

```python
_env = Environment(
    loader=FileSystemLoader(str(_TEMPLATES_DIR)),
    autoescape=select_autoescape(enabled_extensions=("xml", "j2"), default=False),
    ...
)
```

Con `autoescape` activo para `.xml` y `.j2`, cualquier valor de input que el usuario meta en `razon_social`, `descripcion`, etc., se escapa antes de salir al XML. Las entidades XML que pudieran inyectarse (`<!ENTITY x SYSTEM "...">`) se convierten en texto literal (`&lt;!ENTITY ...`), y el parser nunca las procesa como entidades reales.

Se documenta como Info para dejar evidencia de que se evaluó y por qué no se subió de severidad.

Recomendación de hardening defensivo: importar `defusedxml.lxml` y usar `defusedxml.lxml.fromstring` también en estos sitios. Costo es cero (mismo perfil de performance), y blinda contra el escenario de "mañana alguien desactiva autoescape para arreglar un bug de templating y olvida que esa decisión también desnudó XXE acá".

### I2 — Mismatch de versión `pyproject.toml` vs tag

**Severidad:** Info.
**Estado:** Resuelto incidentalmente con el bump a 0.4.1 (pyproject + CHANGELOG + tag futuro alineados).
**Archivo:** `pyproject.toml:7`, tag `v0.4.0`, `CHANGELOG.md:11`.

`pyproject.toml` declara `version = "0.3.0"`. El commit `8b928f4` se titula "feat!: rename a sunat-py 0.4.0" y el tag más reciente es `v0.4.0`, pero el bump del `version` field no se hizo (o se revirtió). El changelog top entry sigue siendo `[0.3.0] - 2026-05-11`.

Resultado posible: o PyPI tiene `0.3.0` y el tag `v0.4.0` es prematuro, o PyPI tiene `0.4.0` y el repo está stale. Ambos escenarios son cosméticos pero confunden el versionado para usuarios externos que comparan `pip show sunat-py` contra el repo.

**Acción sugerida:** decidir cuál es la verdad y alinear los tres lados (pyproject, CHANGELOG, tag).

Surface pública grande. Cualquier refactor de un símbolo del `__all__` es breaking para usuarios PyPI. No es seguridad, es API hygiene — revisar antes de 1.0 si hay funciones internas que se filtraron al top-level.

### I4 — `unpack_cdr` solo lee el primer `.xml` del ZIP

**Severidad:** Info.
**Archivo:** `sunat/packager.py:30-32`.

```python
for name in zf.namelist():
    if name.endswith(".xml"):
        return zf.read(name)
```

Loop con `return` adentro: devuelve el primer match. SUNAT hoy manda siempre un solo XML en el CDR, pero si algún día empieza a mandar varios, el SDK los descarta silenciosamente. Edge case puro, no es bug.

## Áreas verificadas sin hallazgo

Para documentar cobertura, esto es lo que se buscó y **no** se encontró en `packages/core/src/sunat_py/`:

- **Sin `pickle`, `eval`, `exec`, `marshal`, `compile`, ni `yaml.load(...)` sin `SafeLoader`.** No hay puntos de deserialización insegura.
- **Sin `subprocess`, `os.system`, `os.popen`, ni `shell=True`.** El SDK no invoca procesos externos.
- **Sin `verify=False` ni `ssl.CERT_NONE` ni `urllib3.disable_warnings`.** TLS verification queda en default (activa).
- **Sin credenciales hardcodeadas.** Ni en código, ni en wsdl/, ni en templates.
- **Sin `random.random()` / `random.randint()` para nonces o tokens.** No se genera material criptográfico vía RNG débil. El único `hashlib` es `sha256` para el `hashZip` del GRE (correcto).
- **Sin `logging.*` ni `print()` en código de producción.** El SDK no introduce vectores de leak por logs propios. El riesgo es exclusivamente downstream del caller (urllib3 DEBUG, zeep DEBUG).
- **Algoritmos XML-DSig correctos:** `rsa-sha256`, digest `sha256`, canonicalización exclusiva `xml-exc-c14n#`, método `enveloped`. Todos cumplen el estándar SUNAT.
- **Cert RSA enforced.** `cert_loader.py:49-50` rechaza DSA y EC con `ValueError` antes de devolver `CertBundle`.
- **`models.py` sin secretos.** Los dataclasses de input (`InvoiceInput`, `Party`, etc.) no contienen campos sensibles que puedan filtrar por `__repr__`.
- **Validators sin side effects.** `validators/*.py` lee input y lanza `ValidationError`, no logguea, no escribe disco, no hace IO.
- **`ValidationError` no acarrea valores sensibles** por sí mismo — el mensaje lo construye el caller del validator y los mensajes existentes son sobre números de documento y formatos, no sobre passwords ni keys.

## Recomendaciones generales

Más allá de cada finding individual:

1. **Antes de promocionar más el paquete a la comunidad**, arreglar A1 (`CertBundle.__repr__`). Cambiarlo después tiene costo de breaking aunque la API funcional siga igual.
2. **Bajar A2** (`xml_huge_tree=True` + `strict=False`) en el mismo release si se puede. Es un cambio de una línea sin riesgo funcional con CDR reales (que pesan < 50 KB).
3. **Agregar sección "Caller obligations"** al README del SDK, donde se explicite:
   - No habilitar `logging.DEBUG` de urllib3/zeep en producción.
   - No serializar el `zeep.Client` (no pickle, no cache distribuido).
   - No persistir el bearer token del GRE sin cifrar.
   El `SECURITY.md` ya cubre el modelo de amenazas del microservicio. Falta el del SDK como pieza de PyPI.
4. **Lockfile + trusted publishing PyPI** (B3, B4) en el siguiente sprint de "infra del repo". No bloquea nada pero asienta el supply chain.
5. **No introducir `defusedxml`** como dep todavía. Con autoescape de Jinja2 + `huge_tree=False` (post-fix A2) el riesgo XXE es cero. Agregar la dep solo si se vuelve a habilitar `huge_tree` o si se acepta XML externo en algún flujo nuevo.

## Cierre

El SDK no tiene vulnerabilidades remotas explotables hoy. Lo que tiene es **higiene débil alrededor de material sensible** que ya cruzó el paquete: la `key_pem` se filtra al primer `print`, los flags de parser XML están en el modo menos seguro disponible, y los credenciales OAuth2 no tienen ninguna defensa contra el patrón más común de debugging (`logging.DEBUG`).

Cuatro fixes en `cert_loader.py:20`, `client.py:73`, y un docstring + logger filter opcional para `gre_client.py` cierran los cuatro Altos. Ninguno es invasivo; ninguno cambia la firma pública del SDK; todos caben en un solo PR de < 80 líneas.
